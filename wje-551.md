## 项目主要需求

### 一、项目概述

供应链与物流协同平台（SupplyChain Hub）是一个面向中小企业的供应链全流程管理系统，帮助企业管理供应商、仓库、库存和物流运单，实现供应链可视化和协同管理。系统核心是围绕"供应商-仓库-运单"三大业务实体构建全链路管理能力，让采购、仓储、物流三大环节的信息实时互通。

### 二、核心业务实体

#### 2.1 供应商（Supplier）

**数据字段**：
- `id`：主键，UUID
- `name`：供应商名称，必填，唯一
- `contact`：联系人姓名
- `phone`：联系电话
- `email`：联系邮箱
- `address`：详细地址
- `status`：状态，枚举 SupplierStatus（ACTIVE / INACTIVE / BLACKLISTED / PENDING_REVIEW），默认 PENDING_REVIEW
- `rating`：评分，1-5 分，支持小数点一位
- `createdAt`：创建时间
- `updatedAt`：更新时间

**业务关系**：
- 一个供应商可以创建多个采购单（PurchaseOrder），通过 `supplierId` 关联
- 一个供应商可以关联多个运单（Shipment），通过运单的 `supplierId` 关联

**业务动作**：
1. **创建供应商**：表单提交，必填 name + contact + phone，初始状态为 PENDING_REVIEW
2. **审核供应商**：管理员审核通过/拒绝，审核通过后状态变为 ACTIVE，拒绝变为 INACTIVE
3. **评分供应商**：根据历史合作情况对供应商进行 1-5 分评分，支持多次评分取加权平均
4. **禁用/拉黑供应商**：将供应商状态改为 INACTIVE 或 BLACKLISTED，黑名单供应商的运单自动标记异常

**校验规则**：
- 供应商名称不可重复
- 处于 BLACKLISTED 状态的供应商不能创建新的运单
- 评分只能由具有采购经理（PURCHASE_MANAGER）或管理员（ADMIN）角色的用户操作

#### 2.2 仓库与库存（Warehouse + Inventory）

**仓库数据字段**：
- `id`：主键，UUID
- `name`：仓库名称，必填
- `address`：仓库地址
- `capacity`：仓库最大容量（整数）
- `status`：仓库状态（ACTIVE / INACTIVE）
- `createdAt` / `updatedAt`

**库存数据字段**：
- `id`：主键，UUID
- `warehouseId`：所属仓库 ID，外键
- `skuId`：SKU 编码，必填
- `skuName`：SKU 名称
- `quantity`：当前库存数量，整数，>= 0
- `safetyStock`：安全库存阈值，整数，>= 0
- `alertLevel`：库存预警级别，枚举 InventoryAlertLevel（NORMAL / LOW / CRITICAL），由 quantity 与 safetyStock 的关系自动计算
- `updatedAt`：最后更新时间

**业务关系**：
- 一个仓库包含多个库存条目（一对多，通过 warehouseId 关联）
- 库存通过 skuId 可以在多个仓库之间存在（多对多语义，通过不同仓库的相同 skuId 体现）

**业务动作**：
1. **入库操作**：选择仓库 + SKU + 数量，增加库存数量，记录操作日志（审计中间件自动记录）
2. **出库操作**：选择仓库 + SKU + 数量，减少库存数量，出库数量不能超过当前库存
3. **仓库间调拨**：从源仓库出库 + 目标仓库入库，事务性操作，要么全部成功要么全部回滚
4. **库存盘点**：对某个仓库的所有库存条目进行盘点，允许修正实际数量，盘点前后均记录操作日志

**自动预警规则**：
- 当 `quantity <= safetyStock * 0.5` 时，alertLevel 设为 CRITICAL
- 当 `quantity <= safetyStock` 时，alertLevel 设为 LOW
- 当 `quantity > safetyStock` 时，alertLevel 设为 NORMAL
- 预警数据在仪表盘实时展示

#### 2.3 运单（Shipment）

**数据字段**：
- `id`：主键，UUID
- `orderNo`：运单编号，自动生成（格式：SHIP-YYYYMMDD-XXXX），唯一
- `supplierId`：供应商 ID，外键
- `warehouseId`：目标仓库 ID，外键
- `status`：运单状态，枚举 ShipmentStatus（PENDING / SHIPPED / IN_TRANSIT / DELIVERED / EXCEPTION / CANCELLED）
- `trackingNo`：物流追踪号
- `carrier`：承运方名称
- `estimatedArrival`：预计到达时间
- `actualArrival`：实际到达时间
- `remark`：备注
- `createdAt` / `updatedAt`

**业务关系**：
- 一个运单关联一个供应商（通过 supplierId）
- 一个运单关联一个目标仓库（通过 warehouseId）
- 一个运单可以包含多个运单明细（ShipmentItem），每条明细包含 skuId、skuName、quantity

**业务动作**：
1. **创建运单**：选择供应商 + 目标仓库 + 填写运单明细，状态默认 PENDING
2. **发货**：填写 trackingNo + carrier，状态从 PENDING 变为 SHIPPED
3. **在途更新**：状态从 SHIPPED 变为 IN_TRANSIT
4. **签收**：状态变为 DELIVERED，记录 actualArrival，自动触发入库操作（将运单明细中的数量加到目标仓库的对应 SKU 库存中）
5. **异常标记**：任何非终态的运单都可以标记为 EXCEPTION，需要填写异常原因
6. **取消运单**：只有 PENDING 状态可以取消，变为 CANCELLED

**状态流转规则**：
- PENDING → SHIPPED → IN_TRANSIT → DELIVERED（正常流程）
- PENDING / SHIPPED / IN_TRANSIT → EXCEPTION（异常标记）
- EXCEPTION → SHIPPED / IN_TRANSIT（异常解除后恢复）
- PENDING → CANCELLED（取消）

### 三、核心页面需求

#### 3.1 供应链总览仪表盘（/dashboard）

**页面描述**：供应链全局概览，帮助管理者快速了解供应链运行状态。

**展示内容**：
1. **顶部统计卡片**（4 个）：
   - 在途运单数（status 为 SHIPPED 或 IN_TRANSIT 的运单计数）
   - 待签收运单数（status 为 IN_TRANSIT 的运单计数）

   - 左侧：运单状态分布饼图（按 ShipmentStatus 分组计数）

   - 最近 10 条运单列表（按 createdAt 倒序），显示 orderNo、供应商名称、状态、创建时间
   - 库存预警列表（alertLevel 为 LOW 或 CRITICAL 的条目），显示仓库名、SKU、当前数量、安全库存

#### 3.5 共用组件明细

1. **StatusBadge**：根据状态枚举值显示不同颜色的标签。被运单列表、运单详情、仪表盘使用。支持的枚举类型：ShipmentStatus、SupplierStatus、InventoryAlertLevel。每种状态对应固定颜色映射（PENDING=灰色、SHIPPED=蓝色、IN_TRANSIT=橙色、DELIVERED=绿色、EXCEPTION=红色、CANCELLED=灰色等）。

3. **WarehouseSelector**：仓库选择下拉组件，支持单选和多选模式。被库存页面（单选查看库存）和运单页面（选择目标仓库）使用。自动从后端加载仓库列表，支持搜索。

4. **EmptyState**：空状态占位组件，当列表无数据时显示，支持自定义图标和文字。

### 四、横切关注点

#### 4.1 认证授权 + RBAC 权限

**角色定义**：
- `ADMIN`：系统管理员，拥有所有权限
- `PURCHASE_MANAGER`：采购经理，管理供应商和运单
- `WAREHOUSE_MANAGER`：仓库经理，管理仓库和库存
- `VIEWER`：只读用户，只能查看数据

**权限矩阵**：

| 功能模块 | ADMIN | PURCHASE_MANAGER | WAREHOUSE_MANAGER | VIEWER |
|---------|-------|-----------------|-------------------|--------|
| 供应商 CRUD | 全部 | 全部 | 只读 | 只读 |
| 仓库/库存 CRUD | 全部 | 只读 | 全部 | 只读 |
| 运单 CRUD | 全部 | 全部 | 部分（仅签收） | 只读 |
| 仪表盘 | 全部 | 全部 | 全部 | 只读 |
| 审计日志 | 全部 | 只读 | 只读 | 不可见 |
| 用户管理 | 全部 | 不可见 | 不可见 | 不可见 |

**实现要求**：
- DB 层：`role` 表 + `user_role` 关联表 + `permission` 表 + `role_permission` 关联表
- 后端：AuthMiddleware（JWT 验证） + RoleGuard（角色校验） + PermissionGuard（权限校验）
- 前端：路由守卫（router.beforeEach 校验登录状态和页面权限） + v-permission 自定义指令（按钮级权限控制）
- 前端 constants/permissions.ts 定义所有权限标识符，与后端保持一致

#### 4.2 操作日志/审计

**审计范围**：以下写操作必须记录审计日志：
- 供应商：创建、审核、评分、状态变更
- 库存：入库、出库、调拨、盘点
- 运单：创建、发货、签收、异常标记、取消
- 用户：角色变更

**审计日志字段**：
- `id`：主键
- `userId`：操作人 ID
- `username`：操作人名称（冗余，便于查询）
- `action`：操作类型（CREATE / UPDATE / DELETE / STATUS_CHANGE）
- `module`：模块名（SUPPLIER / INVENTORY / SHIPMENT / USER）
- `targetId`：操作目标 ID
- `targetName`：操作目标名称（冗余）
- `detail`：操作详情 JSON（包含变更前后数据对比）
- `ip`：操作人 IP
- `createdAt`：操作时间

**实现要求**：
- DB 层：`audit_log` 表
- 后端：AuditLogMiddleware 自动拦截写操作并记录（通过装饰器或中间件实现，不侵入业务代码）
- 前端：审计日志查询页面（/audit-logs），支持按模块、操作人、时间范围筛选

### 六、全局异常处理

#### 6.1 后端

实现统一异常中间件 `ErrorHandlerMiddleware`：
- 捕获所有未处理的异常，返回统一错误响应格式：`{ code: number, message: string, details?: any }`
- 区分业务异常（BusinessException，自定义类，HTTP 400/404）和系统异常（HTTP 500）
- 生产环境隐藏系统异常的堆栈信息
- 所有异常记录到日志文件

#### 6.2 前端

实现 Axios 响应拦截器：
- HTTP 401：清除 token，跳转登录页
- HTTP 403：提示"无权限执行此操作"
- HTTP 400/404/422：展示后端返回的 message
- HTTP 500：展示"服务器异常，请稍后重试"
- 网络错误：展示"网络连接失败，请检查网络"

#### 10.2 索引策略

- `suppliers.name` 唯一索引
- `shipments.order_no` 唯一索引
- `shipments.status` 普通索引（高频筛选）
- `inventories.(warehouse_id, sku_id)` 联合唯一索引
- `audit_logs.(module, created_at)` 联合索引
- `audit_logs.user_id` 普通索引

#### 10.3 初始数据（Seeds）

- 默认管理员账号：admin / admin123
- 4 个默认角色 + 对应权限
- 3 个示例仓库
- 10 个示例供应商（含不同状态）
- 20 条示例库存记录（含不同预警级别）
- 15 条示例运单（覆盖所有状态）

### 十一、非功能需求

1. **代码严禁合并职责到单一文件**：每个文件只承担一个职责。Controller 只做请求分发和参数校验，Service 只做业务逻辑，Model/Entity 只做数据定义。禁止在一个文件中同时定义路由、控制器和业务逻辑。

2. **每个核心实体贯穿全栈**：以运单为例，必须包含以下文件链路：DB migration → shipment.entity.ts → shipments.service.ts → shipments.controller.ts → shipments.routes.ts → shipments.ts (API层) → shipmentStore.ts → Shipments.vue / ShipmentDetail.vue。

3. **前端路由守卫**：未登录用户跳转登录页；无权限用户跳转 403 页面。

4. **API 请求拦截器**：自动携带 Authorization header；统一处理错误响应。

5. **TypeScript 类型定义**：每个实体在 `types/` 目录下有独立的类型定义文件，与后端 Entity 字段保持一致。

6. **响应式布局**：支持最小 1280px 宽度的屏幕，侧边栏可折叠。

## 代码存放规则

这是前后端分离项目。

前端代码放在 `frontend/`：
页面，组件，路由,状态管理,API 请求,类型定义,样式和资源等放在对应的子文件夹

后端代码放在 `backend/`：
数据库脚本放在 `database/`。

## 部署要求

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

需要提供以下文件：
- 根目录 `docker-compose.yml`：编排前端、后端、数据库及所有依赖服务
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=wjescm`，确保 Compose 项目名不依赖目录名
- `frontend/Dockerfile`：前端项目构建与运行镜像（多阶段构建，先构建再用 Nginx 托管）
- `backend/Dockerfile`：后端项目构建与运行镜像
- `frontend/nginx.conf`：Nginx 配置，处理前端路由和 API 反向代理
- `.env.example`：所有环境变量的示例文件，附注释说明；同时包含 `COMPOSE_PROJECT_NAME` 默认值

docker-compose.yml 要求：
- **不要写 `version:` 字段**（Compose V2 已废弃，会产生 `the attribute version is obsolete` 警告）
- 顶层声明 `name: wjescm` 作为兜底，避免目录名为中文时项目名为空
- 所有服务使用 `container_name` 命名容器，便于管理；容器名带 `${COMPOSE_PROJECT_NAME:-wjescm}` 前缀
- 数据库数据使用命名卷持久化（`volumes:` 顶层声明，不要使用绑定挂载到中文路径）
- 前端通过 Nginx 反向代理后端 API（前端代码里调用 `/api`，由 Nginx 转发到 `backend:38301`，不要硬编码 `localhost`）
- 服务间通过 Docker 内部网络通信，服务名即主机名
- 数据库服务配置 `healthcheck`，后端服务通过 `depends_on: { db: { condition: service_healthy } }` 等待数据库就绪
- 后端服务也建议配置 `healthcheck`，前端 `depends_on` 后端健康
- 暴露端口：前端 `38201:80`，后端 `38301:38301`，数据库 `3306:3306`，Redis `6379:6379`
- 所有可变配置（端口、数据库账号密码、JWT 密钥等）通过环境变量注入
- 支持通过 `docker compose up -d` 一条命令启动全部服务

docker-compose.yml 示例结构：

```yaml
name: wjescm

services:
  db:
    image: mysql:8.0
    container_name: ${COMPOSE_PROJECT_NAME:-wjescm}-db
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_ROOT_PASSWORD}
      MYSQL_DATABASE: ${DB_NAME}
      MYSQL_USER: ${DB_USER}
      MYSQL_PASSWORD: ${DB_PASSWORD}
    ports:
      - "${DB_PORT:-3306}:3306"
    volumes:
      - wjescm-db-data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: ${COMPOSE_PROJECT_NAME:-wjescm}-redis
    restart: unless-stopped
    ports:
      - "${REDIS_PORT:-6379}:6379"
    volumes:
      - wjescm-redis-data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: ${COMPOSE_PROJECT_NAME:-wjescm}-backend
    restart: unless-stopped
    environment:
      DB_HOST: db
      DB_PORT: 3306
      DB_USER: ${DB_USER}
      DB_PASSWORD: ${DB_PASSWORD}
      DB_NAME: ${DB_NAME}
      REDIS_HOST: redis
      REDIS_PORT: 6379
      JWT_SECRET: ${JWT_SECRET}
      PORT: 38301
    ports:
      - "${BACKEND_PORT:-38301}:38301"
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:38301/api/v1/health"]
      interval: 10s
      timeout: 5s
      retries: 5

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    container_name: ${COMPOSE_PROJECT_NAME:-wjescm}-frontend
    restart: unless-stopped
    ports:
      - "${FRONTEND_PORT:-38201}:80"
    depends_on:
      backend:
        condition: service_healthy

volumes:
  wjescm-db-data:
  wjescm-redis-data:
```
